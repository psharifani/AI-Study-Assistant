1. Introduction
================================================================================

1.1 Purpose
This Software Requirements Specification (SRS) describes the requirements for the AI Study Assistant: a web application that helps a learner manage study decks, upload study materials, generate and review flashcards, chat with an AI tutor in the context of a deck, take a mock quiz, and view review statistics. The document is intended for course documentation, future maintenance, and as a basis for testing and design.

1.2 Scope
The system is delivered as:
• A browser-based frontend (single-page application) served in development via Vite.
• A backend REST API (FastAPI) using SQLite for persistence and optional file storage for uploaded documents.
• Integration with the OpenAI API for generation, chat, quiz, grading, and image/PDF vision transcription where applicable.

Out of scope for this SRS (unless extended later): multi-user accounts and authentication, cloud hosting specifics, mobile native apps, and institutional LMS integration.

2. Problem Statement
================================================================================

2.1 Current Challenges
Students often need to turn readings (PDF, text, or photos of notes) into reviewable formats. Doing this manually is slow; keeping track of what to review next is error-prone. Generic chat tools are not tied to the learner’s own uploaded material. There is a need for a focused tool that combines material ingestion, AI-assisted generation, structured review (flashcards + SM-2), practice quizzes, and simple progress visibility.

2.2 Solution Overview
The AI Study Assistant lets the user create decks, attach study files (including images for vision-based transcription), generate flashcards and quizzes from extracted text, review flashcards with four-button SM-2 scheduling, converse with a tutor that can use deck context, and run a mock test graded by rules and (for short answers) AI assistance.


================================================================================
3. Target Users
================================================================================

3.1 Primary Users
• University or college students (e.g. psychology, political science, history) who want to study from their own files.
• The system is designed for a single learner on a local or personal setup; there is no multi-tenant account model in the described version.

3.2 User Roles and Permissions
• Single implicit role: the end user has full access to all decks and features exposed by the UI.
• No login, roles, or permission matrix is implemented in the described version; all API calls are trusted from the configured frontend origin.

================================================================================

4.1 System Purpose
Support the study workflow: organize decks → upload material → generate or create flashcards → review on a schedule → optionally learn via chat and quiz → view statistics.

• Learning chat: threaded sessions per deck; AI replies using optional deck text as context.
• Quiz: generate multiple-choice and short-answer questions from material; grade MC deterministically; grade short answers via AI.
• Statistics: review-oriented charts (e.g. upcoming reviews by day, interval distribution) with selectable time windows.

6. Non-Functional Requirements
================================================================================

6.1 Performance Requirements
• NFR-P1: Typical API operations (list decks, get deck, list flashcards) should complete within a few hundred milliseconds on a modern PC with local SQLite.
• NFR-P2: OpenAI-dependent operations (generation, long chat) are bounded by network latency and model speed; the UI should indicate loading state.
• NFR-P3: Large PDF vision fallback may process up to a capped number of pages to control cost and time.

6.2 Usability Requirements
• NFR-U1: The interface shall use clear labels for review buttons and deck actions.
• NFR-U2: Error messages from the API should be surfaced to the user when requests fail.

6.3 Reliability Requirements
• NFR-R1: Study data shall persist in SQLite across restarts.
• NFR-R2: Uploaded files shall be stored under a configurable data directory; failed uploads should not leave inconsistent deck records when the API returns an error.

6.4 Security Requirements
• NFR-S1: The API shall restrict CORS to configured development origins unless changed for deployment.
• NFR-S2: API keys shall reside in server-side environment configuration, not in client source.

6.5 Compatibility Requirements
• NFR-C1: The web UI shall target modern evergreen browsers (Chrome, Edge, Firefox, Safari recent versions).
• NFR-C2: Backend shall run on Python 3 with documented dependencies.

6.6 Maintainability Requirements
• NFR-M1: The backend shall separate routes, schemas, services, and persistence layers for clarity.
• NFR-M2: Automated unit and end-to-end API tests shall exist for core flows (as implemented in the project).

6.7 Portability Requirements
• NFR-P4: The application shall run on Windows/macOS/Linux for development with documented commands for frontend and backend.

• Optional: microphone/camera are not required by the described system.

7.3 Software Interfaces
• Frontend to Backend: JSON over HTTP REST; base URL configured via environment (e.g. VITE_API_PROXY in development).
• Backend to OpenAI: HTTPS JSON API (chat completions / vision as implemented).
• Backend to filesystem: read/write uploads under the configured uploads directory.
• Backend to SQLite: async SQLAlchemy with aiosqlite driver.

7.4 Communication Interfaces
• Protocol: HTTP/HTTPS.
• Typical development setup: frontend dev server (e.g. port 5173) proxying or calling backend (e.g. port 8080).

8. System Architecture
================================================================================

8.1 Architecture Overview
A layered architecture: React UI → REST API (FastAPI) → services (LLM, document extraction, SM-2) → SQLAlchemy models → SQLite; file storage for uploads.

[SCREENSHOT: Optional single architecture diagram — boxes: UI | API | Services | DB | Files | OpenAI]

8.2 Component Architecture
• Frontend: React + TypeScript + Vite; calls API for all data.
• Backend: FastAPI application module with route handlers; Pydantic schemas for validation.
• Services: OpenAI client wrapper for generation, chat, quiz, grading, vision transcription; document extraction (PDF, images); SM-2 application for reviews.
• Persistence: SQLAlchemy ORM models; async sessions.

8.3 Data Flow
1) User action in UI → HTTP request → route handler → database and/or OpenAI → JSON response → UI update.
2) File upload → stream to disk → extract/transcribe text → store in document row.

10. Database Design
================================================================================

10.1 Database Schema
Logical tables (SQLite):

documents
• id (PK), name, filename, stored_path, content_text, created_at

flashcards
• id (PK), document_id (FK → documents.id), front, back, sort_order
• sm2_ease_factor, sm2_interval_days, sm2_repetitions, sm2_next_review_at, created_at, source_document_name (optional)

chat_sessions
• id (PK), document_id (FK), title, created_at, updated_at

chat_messages
• id (PK), session_id (FK), document_id (FK), role, content, created_at

================================================================================
12. Security Requirements
================================================================================

12.1 Authentication Security
The described system does not implement user authentication; it is suitable for single-user local use. Any network-exposed deployment should add authentication and TLS.

12.2 Authorization Security
No role-based access control; all endpoints assume a trusted client origin.

12.3 Data Security
• API keys must not be committed to source control; use environment files excluded by .gitignore.
• User study content is stored locally in SQLite and on disk; users are responsible for device security.

12.4 Input Validation
• Pydantic models validate JSON bodies (deck names, flashcard fields, chat messages, quiz payloads).
• File uploads use server-side validation of presence and safe filename handling.
