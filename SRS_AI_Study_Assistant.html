1. Introduction

1.1 Purpose

This Software Requirements Specification (SRS) describes the requirements for the AI Study Assistant: a web application that helps a learner manage study decks, upload study materials, generate and review flashcards, chat with an AI tutor in the context of a deck, take a mock quiz, and view review statistics. The document is intended for course documentation, future maintenance, and as a basis for testing and design.

1.2 Scope

The system is delivered as:

  • Browser-based frontend (single-page application) served in development via Vite.
  • Backend REST API (FastAPI) using SQLite for persistence and optional file storage for uploaded documents.
  • Integration with the OpenAI API for generation, chat, quiz, grading, and image/PDF vision transcription where applicable.

Out of scope (unless extended later): multi-user accounts and authentication, cloud hosting specifics, mobile native apps, and institutional LMS integration.

2. Problem Statement

2.1 Current Challenges

Students often need to turn readings (PDF, text, or photos of notes) into reviewable formats. Doing this manually is slow; keeping track of what to review next is error-prone. Generic chat tools are not tied to the learner’s own uploaded material. There is a need for a focused tool that combines material ingestion, AI-assisted generation, structured review (flashcards + SM-2), practice quizzes, and simple progress visibility.

2.2 Solution Overview

The AI Study Assistant lets the user create decks, attach study files (including images for vision-based transcription), generate flashcards and quizzes from extracted text, review flashcards with four-button SM-2 scheduling, converse with a tutor that can use deck context, and run a mock test graded by rules and (for short answers) AI assistance.

3. Target Users

3.1 Primary Users

  • University or college students (e.g. psychology, political science, history) who want to study from their own files.
  • The system is designed for a single learner on a local or personal setup; there is no multi-tenant account model in the described version.

3.2 User Roles and Permissions

  • Single implicit role: the end user has full access to all decks and features exposed by the UI.
  • No login, roles, or permission matrix is implemented in the described version; all API calls are trusted from the configured frontend origin.

4. System Overview

4.1 System Purpose

The AI Study Assistant is a full-stack web application. Its main goals are:

  1. Organize decks and material: Let learners create decks, upload or replace study documents (PDF, text/Markdown, images), and keep content tied to each deck.
  2. Generate and practice: Support manual or AI-generated flashcards, mock quizzes, and a learning chat that can use deck text as optional context.
  3. Schedule reviews: Apply SM-2 scheduling with four review actions (again, hard, good, easy) so cards have sensible next review times.
  4. Show progress: Provide statistics views (e.g. upcoming reviews, interval distribution) with selectable time ranges.

4.2 System Context

[SCREENSHOT: Simple diagram — Student → Browser (React) → HTTP → FastAPI → SQLite / local files; Backend → HTTPS → OpenAI API.]

The major technical components are:

  • Frontend: React + TypeScript + Vite (single-page application).
  • Backend: Python FastAPI REST API (ASGI, Uvicorn).
  • Database: SQLite (relational data via SQLAlchemy async).
  • File storage: Local uploads directory under configurable data path.
  • External AI: OpenAI HTTP API for generation, chat, quiz, grading, and vision transcription.

Narrative: The user interacts only with the web UI. The UI calls the backend REST API. The backend stores data in SQLite and files on disk, and calls OpenAI when generation, chat, quiz, grading, or vision transcription is required.

4.3 Key Features

  • Deck management: Create, rename, delete, list; upload or replace study documents; remove document without deleting the deck.
  • Material extraction: PDF text extraction (with vision fallback for sparse text), plain text/Markdown, image upload transcribed via vision model.
  • Flashcards: Manual create/edit/delete; AI generation from deck material; SM-2 review (again / hard / good / easy).
  • Learning chat: Threaded sessions per deck; AI replies using optional deck text as context.
  • Quiz: Generate multiple-choice and short-answer questions; grade MC deterministically; grade short answers via AI.
  • Statistics: Review-oriented charts with selectable time windows.

6. Non-Functional Requirements

6.1 Performance Requirements

  • NFR-P1: Typical API operations should complete within a few hundred milliseconds on a modern PC with local SQLite.
  • NFR-P2: OpenAI operations are bounded by network and model speed; the UI shall indicate loading state.
  • NFR-P3: PDF vision fallback may cap pages to control cost and time.

6.2 Usability Requirements

  • NFR-U1: Clear labels for review buttons and deck actions.
  • NFR-U2: Surface API errors to the user when requests fail.

6.3 Reliability Requirements

  • NFR-R1: Study data persists in SQLite across restarts.
  • NFR-R2: Failed uploads should not leave inconsistent deck records when the API returns an error.

6.4 Security Requirements

  • NFR-S1: CORS restricted to configured origins unless changed for deployment.
  • NFR-S2: API keys in server-side environment only.

6.5 Compatibility Requirements

  • NFR-C1: Modern evergreen browsers.
  • NFR-C2: Python 3 with documented dependencies.

6.6 Maintainability Requirements

  • NFR-M1: Separation of routes, schemas, services, persistence.
  • NFR-M2: Automated unit and API end-to-end tests for core flows.

6.7 Portability Requirements

  • NFR-P4: Runs on Windows/macOS/Linux for development with documented commands.

7.1 User Interfaces

The client shall provide: deck picker and workspace; flashcard study with four review actions; learning chat with sessions and messages; quiz UI; statistics with range selection.

7.2 Hardware Interfaces

Standard PC or laptop (keyboard, pointer, display). Microphone/camera not required.

7.3 Software Interfaces

  • Frontend ↔ Backend: JSON over HTTP REST (e.g. VITE_API_PROXY in dev).
  • Backend ↔ OpenAI: HTTPS JSON API.
  • Backend ↔ Filesystem: uploads under configured directory.
  • Backend ↔ SQLite: async SQLAlchemy + aiosqlite.

7.4 Communication Interfaces

HTTP/HTTPS. Typical dev: frontend port 5173, backend 8080 (example).
